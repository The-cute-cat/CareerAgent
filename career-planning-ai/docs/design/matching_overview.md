# 岗位处理与匹配核心模块分析

本文档围绕项目中的三大核心能力展开：**岗位插入与清洗、岗位合并与画像生成、岗位匹配与分析**。这三部分并不是彼此独立的功能点，而是构成了一条完整的岗位智能处理链路：系统先把多来源、异构格式的原始岗位数据清洗并入库，构建统一的原始岗位数据层；再基于语义向量与聚类算法，将碎片化岗位归并为标准岗位画像，并沉淀为可复用的岗位知识库；最后结合学生画像，通过多向量混合召回与大模型深度分析，输出可解释的人岗匹配结果与改进建议。  

从系统架构角度看，这条链路本质上完成了三个层次的能力升级：第一步解决的是**原始岗位数据如何标准化接入**的问题，核心产物是结构统一、可继续加工的 `job_original` 原始岗位库；第二步解决的是**如何从大量相似但表达不一致的岗位中抽取标准岗位知识**的问题，核心产物是 `job_profile` 标准岗位画像库及其向量索引；第三步解决的是**如何把岗位知识真正转化为面向用户的推荐与诊断能力**的问题，核心产物则是推荐列表、差距矩阵、行动建议和单岗位诊断报告。三者层层递进，分别对应“数据治理层—知识抽象层—智能应用层”的系统演化过程。  

因此，这份文档的分析重点不只是介绍单个脚本或单个接口，而是说明项目如何通过**数据清洗、语义聚类、结构化建模、向量检索和大模型分析**等技术手段，把原始招聘信息逐步转化为能够支撑职业规划、岗位推荐和能力诊断的智能系统能力。对比赛答辩而言，这三步也恰好构成了一条清晰的技术主线：**先有数据底座，再有岗位知识，再有用户侧智能匹配**。

## ⭐️ 第一步：岗位插入与岗位清洗 (Job Insertion & Cleaning)

**本部分使用的方法（一句话）：** 采用“多格式文件发现 + 分渠道清洗标准化 + 结构化映射 + 异步批量入库”的分层流水线，将来源不一、字段不齐、格式混乱的原始岗位数据转化为统一的原始岗位库。

**核心职责：** 该阶段承担整个岗位智能分析链路的“数据底座建设”任务。系统先从本地目录批量发现 Excel / CSV 岗位文件，再按文件类型分流到对应清洗器，对岗位名称、岗位描述、薪资、地点、日期等关键字段做规范化处理，随后映射为 `JobInfo` 实体并异步写入 MySQL 的 `job_original` 表。这个阶段的目标不是直接生成岗位画像，而是先构建一个**字段统一、噪声受控、可继续聚类与向量化**的原始岗位数据层，为后续“岗位合并”“岗位画像生成”“人岗匹配”提供稳定输入。

**主要代码文件：**

- `job_insert.py`
- `job_table_cleaning_csv.py`
- `job_table_cleaning_excel.py`
- `job_info.py`
- `job_info_repository.py`

### 1.1 所用工具与技术栈

1. **Pandas 数据清洗引擎**
   该模块大量依赖 Pandas 完成 Excel / CSV 的表格读取与列级处理。它不仅承担基础 I/O，还负责对原始招聘字段做字符串标准化、缺失值兜底、列存在性宽容判断、重复数据删除、排序、重命名与导出保存，是整个清洗链路的核心执行层。尤其在 Excel 场景下，`岗位详情`、`岗位名称`、`薪资范围`、`更新日期` 等字段均通过 Pandas 列操作完成统一化处理；在 CSV 场景下，岗位描述重组、地址拼接、字段删减与列重命名也全部依赖 DataFrame 流式处理实现。
2. **Path / pathlib 文件系统遍历机制**
   `job_insert.py` 中通过 `Path` 与 `rglob()` 实现对指定目录的递归扫描，统一收集待导入岗位文件，并结合后缀名自动判定走 Excel 处理链还是 CSV 处理链。这种做法的优势在于：一方面屏蔽了平台差异，另一方面把“文件发现”从“文件处理”中解耦出来，便于后续替换数据目录、扩展更多文件类型或加入分批导入策略。
3. **SQLAlchemy 异步 ORM + aiomysql 驱动**
   数据库存储层采用 `create_async_engine`、`AsyncSession`、`async_sessionmaker` 构建异步会话工厂，再由仓储层 `JobRepository` 统一封装批量创建、单条创建、查询、分页、更新与删除能力。批量导入时通过 `create_all(job_list)` 将清洗后的 `JobInfo` 实体整体提交到数据库，并在提交后逐条 `refresh` 获取数据库回填字段。该设计使“数据清洗”和“数据落库”边界清晰，便于后续复用到聚类、回写画像 ID 等场景。
4. **asyncio 并发调度与限流控制**
   在目录级批量导入场景下，系统通过 `asyncio.gather` 并发调度多个文件导入任务，并使用 `asyncio.Semaphore(max_concurrent)` 控制最大同时处理文件数，避免因为瞬时打开过多文件或提交过多数据库事务而导致连接池耗尽、内存抖动或吞吐异常。这种“外层并发、内层限流”的设计对于岗位数据批量初始化特别重要，因为导入任务天然属于 I/O 密集型场景。
5. **日志与异常观测机制**
   整条链路中大量使用 `log.info / warning / error` 记录文件识别、原始行数、去重数量、批次执行状态、字段缺失、单行解析失败、落库失败等信息。这使岗位导入不仅是一个“能跑”的脚本，而是一个具备可观测性的工程化数据入湖流程，便于后续在比赛演示、测试排障和数据治理中快速定位问题。
6. **可选的 LLM 岗位预过滤能力**
   Excel 清洗脚本中还实现了 `filter_computer_jobs_excel()`，它会先提取岗位名称、去重、再调用 `ChatTongyi + PydanticOutputParser` 对岗位名做“是否属于计算机 / 软件相关岗位”的批量判断，并按结果反向过滤原表。虽然该能力当前不是 `process_and_insert_jobs()` 的主链默认步骤，但从工程设计上看，它提供了一种“在入库前先做领域筛选”的能力，可用于提高技术岗位数据集纯度，降低后续聚类与画像阶段的噪声。

### 1.2 数据模型与输入输出约束

1. **输入侧：多源异构岗位文件**
   当前导入主流程直接支持三类后缀：`.xls`、`.xlsx`、`.csv`。其中 Excel 文件更偏向标准化列式招聘数据，CSV 文件则包含更多渠道字段、历史字段命名不统一甚至拼写异常的情况，例如 `describtion` 这类原始列名也被保留并兼容处理。系统不会假设所有数据源字段一致，而是以“存在则处理，不存在则告警”的宽容模式执行。
2. **中间层：统一映射为 `JobInfo` 实体**
   Excel 侧通过 `EXCEL_TO_MODEL_MAPPING` 将“岗位名称、地址、薪资范围、公司名称、所属行业、公司规模、公司类型、岗位编码、岗位详情、更新日期、公司详情、岗位来源地址”等列映射到模型字段；CSV 侧通过 `CSV_TO_MODEL_MAPPING` 将 `job / salary / url / describtion / update_time` 等字段映射为统一的 `job_title / salary_range / job_source_url / job_desc / updated_time`。这一层的关键价值在于把不同来源的字段命名统一进数据库域模型。
3. **输出侧：写入 `job_original` 原始岗位表**
   `JobInfo` 模型对应数据库表 `job_original`，核心字段包括 `job_title`、`location`、`salary_range`、`company_name`、`company_size`、`company_type`、`industry`、`company_desc`、`job_desc`、`job_source_url`、`updated_time` 等；同时保留 `job_profile_id` 作为后续岗位合并后回写标准岗位画像的关联键，并通过 `is_deleted` 实现逻辑删除。也就是说，第一步的产物不是最终岗位画像，而是一个可被后续聚类和画像模块继续消费的“原始岗位标准层”。

### 1.3 实现思路与处理流程

1. **目录扫描与任务装配：批量发现待处理文件**
   系统首先调用 `get_all_files(folder_path, recursive=True)` 扫描指定岗位数据目录，获取所有文件的绝对路径。该函数会先校验路径是否存在、是否为目录，再根据是否递归决定使用 `rglob("*")` 还是 `iterdir()`。这一步实现的是“待处理文件清单构建”，为后续异步调度提供输入队列。
2. **按后缀路由到不同清洗链**
   单文件处理入口是 `process_and_insert_jobs(file_path, max_process=None)`。该函数先对路径存在性做校验，再根据后缀执行路由：
   - `.xls / .xlsx` 走 `clean_job_excel()` + `read_excel_to_jobinfo()`；
   - `.csv` 走 `clean_job_csv()` + `read_csv_to_jobinfo()`；
   - 其他格式直接抛出 `ValueError`。
     这说明系统采用的是“先清洗、后建模、再入库”的两段式处理，而不是把所有清洗逻辑塞进入库函数里，有利于保持代码职责单一。
3. **Excel 清洗链：面向标准列式招聘表的规范化处理**
   `clean_job_excel()` 的处理更偏向“对结构较完整的招聘表做规范化清理”，其核心能力包括：
   - 仅允许 `.xls / .xlsx`，提前拦截非法输入；
   - 读取原始 Excel 后记录原始行数；
   - 对 `岗位详情` 字段删除 `<br>`、换行、制表符和多余空格，减少后续文本脏噪声；
   - 对 `岗位名称` 执行括号内容剥离，去掉诸如“前端开发（急聘）”“Java工程师(校招)”中的附属营销信息；
   - 对薪资与日期做标准化处理，尽量把多样写法收敛为统一格式；
   - 可选执行去重；
   - 最终覆盖原文件保存。
     换句话说，Excel 链路的设计目标是把“人工采样表 / 标准导出表”变成适合直接映射实体的数据。
4. **CSV 清洗链：面向渠道字段杂乱场景的结构重组**
   `clean_job_csv()` 的职责更强，不仅做“清理”，还做“结构重组”。它包含几类关键处理：
   - 对 `job` 执行岗位名称清洗，去掉括号附加信息；
   - 对 `salary` 执行薪资标准化，支持 `1.6-3万·14薪`、`5000-10000元·13薪`、`100-150元/天` 等多种表达，统一换算为月薪或年薪文本；
   - 对 `update_time` 做日期归一化；
   - 把 `city + district` 合并为统一的 `location`；
   - 用 `experience + education + skills_item + describtion` 重组生成 `job_desc`，把原本分散在多列的信息合成为更完整的岗位描述文本；
   - 将原始列重命名为统一模型字段名；
   - 删除不再需要的中间列，如 `job_type`、`headcount`、`employer_tag`、`city`、`district`、`education`、`skills_item`、`experience`、`describtion`、`address`；
   - 按 URL 或标题组合去重，并按岗位名称稳定排序。
     这说明 CSV 链路本质上不仅是“清洗”，更是“半结构化招聘字段向统一岗位对象的重构过程”。
5. **字段级标准化方法：将脏数据转为可计算文本**
   在 CSV 工具层中，系统为关键字段单独实现了多个函数，体现出较强的数据治理意识：
   - `normalize_text()`：将空值、NaN、空字符串统一折叠为 `None`；
   - `clean_html_and_spaces()`：清理描述中的 HTML、换行与冗余空格；
   - `clean_job_name()`：删除岗位名中的括号附加信息；
   - `standardize_salary()`：把复杂薪资表达转换为统一的“元 / 月”或“元 / 年”；
   - `build_location()`：合并城市与区县信息；
   - `build_job_desc()`：拼装经验、学历、技能与职位描述，构造更完整的岗位文本输入。
     这些函数的存在说明系统并非只做字段搬运，而是有意识地把原始招聘数据转化为后续语义分析更友好的表示。
6. **实体映射：由表格行转为 ORM 对象**
   清洗后的数据不会直接以 DataFrame 入库，而是通过 `read_excel_to_jobinfo()` / `read_csv_to_jobinfo()` 逐行组装为 `JobInfo` 实体列表。Excel 侧依赖固定映射表逐列转字段，并尝试解析日期；CSV 侧则在映射基础上额外调用岗位名清洗、薪资归一化、地点合并和描述重组函数。若某一行解析失败，系统会记录日志并跳过，而不是让整批数据导入中断。这种“逐行容错”的策略在真实招聘数据处理中很有必要。
7. **异步批量入库：构建原始岗位数据库**
   当单文件 `job_list` 构建完成后，`process_and_insert_jobs()` 会通过 `AsyncSessionLocal()` 创建异步会话，并调用 `JobRepository.create_all(job_list)` 批量提交到数据库。仓储方法内部通过 `session.add_all()` + `commit()` 完成持久化，并对每条对象执行 `refresh()`，确保模型实例获得数据库生成的主键与时间字段。这一步意味着“原始岗位数据”正式进入系统可查询、可聚类、可回写的数据库层。
8. **并发调度与资源释放：保障批量导入稳定性**
   `main(max_concurrent=5)` 会先获取所有文件路径，再为每个路径封装 `safe_process()` 任务，外层通过 `asyncio.gather()` 并发执行，内层通过 `Semaphore` 做受控并发。无论任务成功还是失败，最终都会在 `finally` 中执行 `engine.dispose()`，主动关闭数据库连接池。这种处理方式体现出比较规范的批处理工程实践：既追求吞吐量，也显式管理资源生命周期。

### 1.4 本阶段的核心功能点

1. **多格式岗位文件统一接入**
   支持 Excel 与 CSV 两类主流招聘数据格式，降低原始数据接入门槛。
2. **岗位基础字段标准化治理**
   对岗位名称、岗位描述、地点、薪资、日期等关键字段做统一清洗，解决多来源数据格式不一的问题。
3. **渠道异构字段向统一领域模型映射**
   通过 Excel / CSV 两套映射关系，把不同命名风格的数据对齐到 `JobInfo` 统一实体。
4. **原始岗位文本增强**
   尤其在 CSV 场景下，通过拼接经验、学历、技能、原始描述形成更完整的 `job_desc`，为后续向量化、聚类和画像提炼提供更高质量文本。
5. **异步批量入库与导入容错**
   通过异步仓储和逐行异常跳过机制，提高大批量岗位导入的稳定性和吞吐效率。
6. **为后续岗位合并预留结构接口**
   原始岗位表中保留 `job_profile_id` 字段，表明该阶段不仅是简单落库，还为下一阶段“原始岗位 → 标准岗位画像”的多对一关联做了结构准备。

### 1.5 本阶段的工程价值

从系统架构角度看，第一步的价值不只是“把文件导进去”，而是在项目最前端建立一个可持续演进的数据治理层。它把原本零散、噪声大、字段不统一的岗位源数据，转化成结构统一、文本可计算、语义可聚类的原始岗位库。没有这一步，后续 HDBSCAN 聚类会受到岗位名脏词和描述噪声影响，LLM 画像提炼也会因为字段缺失与格式混乱而输出不稳定。因此，这一阶段本质上是在为整个职业规划系统构建“原始岗位标准层”。

## ⭐️ 第二步：岗位合并与画像生成 (Job Merging & Profile Building)

**本部分使用的方法（一句话）：** 采用“原始岗位向量同步 + HDBSCAN 语义聚类 + LLM 结构化提炼 + 岗位画像落库 + 向量索引同步”的多阶段流水线，将大量碎片化原始 JD 合并为可复用、可检索、可匹配的标准岗位画像。

**核心职责：** 本阶段是系统从“原始岗位数据层”走向“标准岗位知识层”的关键中枢。其核心目标不是简单做岗位去重，而是先基于原始岗位标题与描述构建语义表示，对岗位进行无监督聚类，再针对每个语义簇调用大模型提炼共性能力要求、职业技能、发展潜力与岗位属性，最终生成统一的标准岗位画像并写入 `job_profile` 表，同时同步至 Milvus 向量库，作为后续人岗匹配模块的岗位知识基座。

**主要代码文件：**

- `job_merger.py`（岗位合并主流程入口）
- `job_profile_builder.py`（基于多条 JD 构建岗位画像）
- `struct_job_txt.py`（岗位画像结构化模型定义）
- `HDBSCAN.py`（岗位聚类主逻辑）
- `job_original_vector_store.py`（原始岗位向量同步与读取）
- `job_vector_store.py`（标准岗位画像向量库存储与检索）
- `job_portrait.py` / `job_portrait_repository.py`（岗位画像表与仓储层）

### 2.1 所用工具与技术栈

1. **HDBSCAN 密度聚类算法**
   当前岗位合并并不是按岗位名称做字符串去重，而是先对岗位文本向量做无监督聚类。聚类模块 `cluster_standard_jobs_with_hdbscan()` 使用 HDBSCAN，对高维岗位语义向量执行自动聚类，并将 `cluster_id = -1` 的样本识别为噪声岗位。同时，代码中还配置了 `min_cluster_size`、`min_samples`、`cluster_selection_method="eom"` 等参数，用于控制簇形成门槛、密度敏感度与最终簇划分策略。这类算法比简单的 KMeans 更适合岗位文本这种簇数未知、分布不均且噪声较多的场景。
2. **本地语义向量模型 + 原始岗位向量库**
   在聚类之前，系统会先通过 `JobOriginalVectorStore` 统一构建“原始岗位向量层”。该模块对 `job_original` 表中的岗位标题和岗位描述进行轻量文本清洗，构造用于聚类的文本特征，再使用本地 `SentenceTransformer` 模型（默认 `BAAI/bge-base-zh-v1.5`）生成向量，并存入 Milvus 集合 `job_original_embeddings`。同时它还带有数据库数量与向量库数量的一致性校验和增量补齐机制，避免每次聚类都重复向量化全量岗位。
3. **PCA 降维 + 向量预处理**
   在 HDBSCAN 正式聚类前，代码会检查向量维度；当向量维度大于 128 时，会先使用 PCA 将其压缩到 128 维，再进入聚类过程。这样做的工程意义在于降低高维稀疏空间下的噪声干扰与聚类计算成本，提高聚类的稳定性和速度。
4. **LLM 结构化岗位画像提炼**
   对每个有效语义簇，系统不会直接把簇中心当结果，而是将该簇内多条 JD 组合成一个岗位集合文本，交给 `JobProfileBuilder` 做“标准岗位提炼”。`job_profile_builder.py` 中的提示词明确要求模型必须基于“全部职位描述集合”进行横向比较，先筛选出最应归并为同一类的岗位，再只输出一个标准岗位对象，并严格满足字段枚举、数组非空、中文命名和结构化 JSON 的约束。这意味着该阶段不是纯摘要，而是“语义归并 + 结构化建模”的复合任务。
5. **Pydantic / 结构化模型约束**
   岗位画像输出并不是自由文本，而是被约束到 `Profiles` 或 `JDAnalysisResult` 等结构化模型中。代码中为基础要求、职业技能、职业素养、发展潜力、岗位属性分别定义了独立的数据模型和枚举字段，并通过兜底模板、空数组修复和字段名修正逻辑，尽量消除大模型输出时的格式漂移问题。这使岗位画像不仅可读，而且能被数据库存储、向量化、匹配引擎和前端组件直接消费。
6. **asyncio 并发调度与受控限流**
   岗位合并主流程中，通过 `_analyze_cluster_with_semaphore()` 对每个有效簇执行并发分析，并用 `asyncio.Semaphore(max_concurrency)` 限制同时调用大模型的簇数量。这里的设计重点是：**LLM 分析可以并发，但数据库写入保持顺序**。这样既能提升岗位画像生成吞吐量，又避免多个协程共享同一 `AsyncSession` 带来的事务竞争和游标问题。
7. **Milvus 标准岗位画像向量库**
   画像生成完成后，系统会将 `job_profile` 表中的画像重新读取出来，转换为 `JDAnalysisResult`，再写入标准岗位向量库 `job_matching_profiles`。这个库不是单向量，而是按岗位画像的不同子模块拆成四个向量字段：`vec_basic`、`vec_skills`、`vec_literacy`、`vec_potential`，同时保留 `degree_level`、`exp_level` 和 `raw_data` 标量字段，便于后续做多维加权召回与硬条件过滤。也就是说，第二步的输出不仅是数据库记录，更是一套可检索的多维岗位知识索引。

### 2.2 数据模型与核心输入输出

1. **输入：原始岗位语义集合**
   第二步的直接输入不是文件，而是第一步已经写入 `job_original` 表的原始岗位对象。每条记录至少包含岗位标题、岗位描述、行业、薪资等信息，作为后续构建文本特征和聚类分析的基础。
2. **中间结果一：岗位语义簇**
   经过原始岗位向量化与 HDBSCAN 聚类后，系统会得到一个 `Dict[int, List[JobInfo]]` 结构，其中 key 为 `cluster_id`，value 为该簇下的岗位列表；其中 `-1` 表示噪声簇。此时系统已经将“原始岗位列表”转化为了“语义相近岗位组”，但还没有生成标准画像。
3. **中间结果二：结构化画像字典**
   每个有效簇会被 `analyze_job_profiles()` 送入 `JobProfileBuilder`。当前主流程调用的是“只输出 `Profiles` 主体”的模式，也就是模型只返回岗位画像主体，而不直接返回完整 `job_id` / `job_name` 包装结构。这个结果随后会被作为 `skills_req` 写入岗位画像表。
4. **输出一：标准岗位画像表 `job_profile`**
   标准岗位画像最终存入 `job_profile` 表，对应模型 `JobPortrait`。其核心字段包括：
   - `job_title`：标准岗位名称
   - `skills_req`：岗位要求画像 JSON
   - `radar_data`：后续可视化或衍生画像数据
   - `tech_skills`、`certificates`、`score_innovation`、`score_learning`、`score_stress`、`score_communication` 等辅助画像字段
   - `is_deleted`、`create_time`、`update_time` 等系统字段
     这张表本质上就是项目的“标准岗位知识库”。
5. **输出二：原始岗位到标准岗位的映射关系**
   在画像入库后，系统会把该簇下的原始岗位记录回写到对应画像 ID 上，形成“多个原始岗位 → 一个标准岗位画像”的多对一归并关系。这个关系使得系统既保留原始岗位颗粒度，又能在上层用统一标准岗位进行分析与推荐。
6. **输出三：标准岗位画像向量索引**
   系统会再从 `job_profile` 表中读取全部画像，把 `skills_req` 转换成 `JDAnalysisResult` 后写入 Milvus 集合 `job_matching_profiles`。该步骤的直接意义是把“岗位画像 JSON”进一步转化为“后续匹配可召回的向量知识库”。

### 2.3 实现思路与处理流程

1. **原始岗位向量同步：构建聚类输入层**
   聚类入口 `cluster_standard_jobs_with_hdbscan()` 并不直接从数据库读取文本后立刻聚类，而是先委托 `JobOriginalVectorStore` 负责原始岗位文本构建、向量化与 Milvus 同步。该模块会检查“数据库岗位数量”与“原始岗位向量集合数量”是否一致：
   - 一致则直接复用现有向量；
   - 数据库多于向量库时，只对新增岗位做增量向量化；
   - 向量库数量异常大于数据库时直接报错。
     这种设计避免了每次岗位合并都做全量重复向量化，属于比较成熟的增量同步思路。
2. **文本特征构建：标题主导、描述辅助**
   原始岗位向量并不是直接拿 `job_desc` 做 embedding，而是通过 `build_text_feature()` 构造聚类文本：以清洗后的岗位标题为主，标题重复一次作为轻量加权，再追加岗位描述，且显式剔除部分噪声词和薪资表达。这样做能让聚类更加关注“岗位职责与能力模型”，而不是被薪资、福利或营销词干扰。
3. **HDBSCAN 聚类：形成候选岗位簇**
   当有效岗位和向量准备完毕后，系统执行 HDBSCAN 聚类。聚类后的结果会按簇编号组织为岗位列表，`-1` 被视为噪声岗位；同时主流程还会继续过滤掉岗位数少于 2 的小簇。也就是说，真正进入岗位画像提炼阶段的，是“非噪声且具备最低规模”的有效岗位簇，而不是所有聚类输出结果。
4. **簇级并发分析：把岗位簇交给大模型归并**
   `job_merger()` 中会先把有效簇收集成 `valid_clusters`，然后为每个簇创建异步任务，调用 `_analyze_cluster_with_semaphore()`。这个函数内部只做一件事：在受控并发下调用 `analyze_job_profiles(jobs)`，把一个岗位簇转换成一个标准岗位画像主体。这里的关键点在于，系统是按“簇”为单位调用模型，而不是按“单条岗位”调用模型，能够更好地提取岗位共性、统一命名与统一要求。
5. **LLM 结构化提炼：从多个 JD 到一个标准岗位对象**
   `JobProfileBuilder` 在提示词层明确要求模型必须综合分析岗位职责、任职要求、技能要求、核心产出、业务场景和工作边界，只有在综合重合度足够高时才允许合并，并且最终只能输出 1 个标准岗位对象。与此同时，代码还通过 `PROFILE_TEMPLATE` 和 `JD_TEMPLATE` 为模型提供明确结构样例，并在后处理阶段修复空数组、字段别名和异常键名。这样做的目的，是尽可能把大模型输出限制在“可控的岗位画像 Schema”内，减少幻觉与字段漂移。
6. **岗位画像落库：建立标准岗位知识层**
   当模型返回画像字典后，主流程会先提取标准岗位名称，构造 `JobPortrait(job_title=..., skills_req=..., is_deleted=0)`，再通过 `JobPortraitRepository.create()` 逐条写入 `job_profile` 表。写入成功后，系统会获取新画像的主键 ID，并回填到当前画像 JSON 中，再更新回数据库，以保证画像数据和数据库主键关联一致。
7. **关系回写：打通原始岗位与标准岗位画像**
   画像保存后，系统会遍历当前簇下的原始岗位集合，把这些岗位与新生成的画像 ID 建立关联。该步骤的本质是构建“岗位归并血缘关系”，为后续从原始岗位追溯到标准岗位、从标准岗位反查来源岗位提供关系基础。
8. **标准岗位向量同步：生成匹配侧可检索索引**
   完成画像落库与关系回写后，主流程还会调用 `store.load_all_jd_results_from_db(session=session)` 重新读取岗位画像，并把 `skills_req` 转成 `JDAnalysisResult`。随后再调用 `store.insert_job_async(stats["items"])` 将这些标准岗位画像向量化后写入 `job_matching_profiles`。这说明第二步并不是“聚类完就结束”，而是显式承担了“为匹配系统准备岗位画像索引”的职责。

### 2.4 本阶段的核心功能点

1. **原始岗位的语义归并**
   解决招聘平台中大量“同岗不同名、同义不同写法、表述颗粒度不同”的问题，把零散 JD 收敛为标准岗位类别。
2. **岗位画像自动生成**
   自动提炼学历要求、经验要求、核心技能、工具平台、语言能力、职业素养、发展潜力、岗位属性等字段，形成统一的岗位画像结构。
3. **标准岗位命名与组织化沉淀**
   系统要求模型输出规范化的标准岗位名，而不是直接复用某条原始岗位名称，从而使岗位知识库更适合做组织架构梳理、画像复用和前端展示。
4. **原始岗位到标准岗位的多对一映射**
   保留“来源岗位 → 标准岗位画像”的归并关系，为后续岗位追踪、数据修正、画像解释与统计分析提供支撑。
5. **匹配侧岗位知识库初始化**
   将岗位画像同步为 Milvus 中的多向量索引，直接服务第三步的人岗匹配召回与重排。
6. **工程化容错与并发保护**
   主流程中对噪声簇、小簇、LLM 返回异常、画像创建失败等都做了显式分支与错误记录，并通过信号量控制模型并发，保证大批量岗位合并时的稳定性。

### 2.5 本阶段的工程价值

从工程角度看，第二步真正完成的是一次“岗位知识抽象”。第一步得到的仍然是渠道化、碎片化的岗位记录，而第二步把这些记录提升为统一的标准岗位画像，并进一步变成可被向量检索、可被学生画像匹配、可被前端展示的岗位知识资产。换句话说，它解决的不是单纯的数据存储问题，而是岗位语义标准化、知识沉淀和后续智能推荐可用性问题。没有这一步，后续匹配系统只能在原始 JD 上做粗粒度相似度计算，很难得到稳定、可解释、可复用的人岗匹配结果。

# ⭐️ 第三步：岗位匹配与分析（Job Matching & Analysis）

## 知识库检索与画像补全 > 硬条件过滤 > 混合召回 > Adj-LightGBM > LLM Top-N 深度重排 > 知识库证据检索 > 结构化差距解释

**本部分使用的方法（一句话）：**
采用“**知识库检索与画像补全 + 硬条件过滤 + 多向量混合召回 + Adj-LightGBM 轻量精排 + LLM Top-N 深度重排 + 知识库证据检索 + 结构化差距解释**”的多阶段匹配方法，输出既可排序、又可解释、还能给出改进建议的人岗匹配结果。

岗位匹配模块的目标并不是只返回一个相似度分数，而是构建一个完整的“**先补全、再筛选、再召回、再排序、再解释**”闭环。系统首先结合岗位知识库与用户画像信息完成基础画像补全，然后依据学历、经验等硬性约束过滤掉明显不适配的岗位；接着基于多维岗位画像向量完成候选岗位召回，并通过 Adj-LightGBM 对候选集合进行轻量学习排序；在此基础上，再调用大模型对 Top-N 候选岗位逐条做深度 Gap 分析与重排序，并从知识库中检索对应岗位要求、技能条目和能力标准作为解释证据，最终形成既适合前端推荐列表展示、又支持用户理解差距和制定提升路径的结构化结果。

从整体架构上看，这一阶段已经不再是单纯的“相似岗位搜索”，而是将**规则过滤、语义召回、学习排序、大模型分析、证据解释**串联起来，形成面向职业规划场景的智能推荐与诊断链路。

## 3.1 所用工具与技术栈

### 1. Pydantic 学生画像建模

匹配流程首先依赖 `StudentProfile` 作为统一输入模型。该模型将学生信息拆分为 `basic_info`、`skills`、`literacy`、`potential`、`constraints`、`experience` 六个一级模块，并对行业知识评分、项目经验评分、沟通能力、学习能力等量化字段设置范围约束。这意味着匹配系统接收的并不是松散的自由 JSON，而是经过结构化约束的学生画像对象，为后续知识补全、向量召回、排序与 LLM 分析提供了稳定输入。

### 2. 知识库检索与画像补全

在正式匹配前，系统会将学生已有信息与岗位知识库中的标准岗位画像、技能标签、能力要求进行关联，用于补全学生画像中不完整或不够标准化的字段。例如，将学生填写的非标准技能描述映射为知识库中的统一技能标签，将项目经历中的能力表达补全为更适合计算与匹配的结构化字段。这样可以减少前端输入差异带来的召回偏差，提升后续匹配的稳定性与泛化能力。

### 3. Milvus 多向量混合检索

岗位召回并不是基于单一向量完成，而是依赖 `job_matching_profiles` 集合中的四个向量字段：`vec_basic`、`vec_skills`、`vec_literacy`、`vec_potential`。系统分别为学生画像的四个对应维度生成向量，再通过 `hybrid_search()` 发起多路 ANN 检索，并使用 `WeightedRanker` 进行加权融合。这样，召回结果不再单纯依赖某一项技能相似度，而是综合考虑基础要求、职业技能、职业素养和发展潜力四个层面。

### 4. 硬条件过滤机制

在正式召回前，系统会先把学历和经验映射为数值等级，例如博士=4、本科=2、专科=1，经验按“应届 / 1-3年 / 3-5年 / 5年以上”映射为 0~3，并构造过滤表达式 `degree_level <= user_degree_level && exp_level <= user_exp_level`。这意味着匹配流程并不是一开始就将所有岗位送入向量召回，而是先排除在学历或经验层面明显不满足要求的岗位，从而减少无意义召回与后续排序、分析成本。

### 5. 阿里云 Embedding 与岗位画像向量化

岗位画像向量库存储阶段与学生召回阶段都依赖 Embedding 模块。`JobVectorStore` 内部使用 `AliyunEmbedding`，将岗位画像的四个子模块分别转换为文本后生成四路向量。标准岗位画像向量库不仅保存向量，还存储 `degree_level`、`exp_level` 和 `raw_data`，因此匹配结果既能参与向量计算，也能直接返回原始岗位画像供后续排序与分析使用。

### 6. Adj-LightGBM 轻量学习排序

在混合召回完成后，系统进一步引入 Adj-LightGBM 作为候选岗位的轻量精排模块。该模块主要面向“召回后的候选集合”，结合多维召回得分、硬条件匹配情况、岗位基础要求、学生技能覆盖度、项目经历相关性等特征，对候选岗位进行二次排序。它的作用不是替代向量召回，而是作为召回层与 LLM 深度分析层之间的“中间排序器”，在保持速度较快的前提下，进一步提升候选集合的质量，减少无效岗位进入大模型分析阶段。

### 7. LLM 深度分析 Agent

经过学习排序后的 Top-N 候选岗位不会直接按分数展示，而是进一步交给 `CareerAnalystAgent` 做深度差距分析。该 Agent 会把学生画像与岗位画像拼接成 Prompt，要求模型输出统一的 JSON 结果，字段包括 `can_apply`、`score`、`missing_key_skills`、`gap_matrix`、`actionable_advice`、`all_analysis`。这一步的核心作用，是把“相似度排序”升级为“可解释的职业判断”。

### 8. 知识库证据检索

在 LLM 完成候选岗位的深度分析后，系统还会进一步从岗位知识库中检索该岗位对应的技能要求、能力标准、典型任务、工具要求或发展路径节点，作为最终解释结果的证据支撑。这样，前端展示的“差距说明”和“改进建议”就不只是模型生成的自然语言结论，而是能够关联回岗位知识库中的结构化依据，提升解释的可信度、可追溯性和展示效果。

### 9. 定向岗位匹配分析器

对于“用户点开一个具体岗位查看详细诊断”的场景，系统单独实现了 `analyze_user_job_match()`。它会根据 `job_id` 查询岗位画像，根据 `user_id` 查询学生画像，将双方画像序列化后送入模型进行分析，最终输出统一的 `UserJobMatchResult` 结构，字段包括总体评价、匹配等级、优势分析、不足分析、改进建议、面试风险点和最终建议。

### 10. 异步并发与线程池降级

`CareerAnalystAgent` 当前采用 `asyncio.gather()` 并发分析多个候选岗位，同时用 `asyncio.to_thread()` 把同步模型调用包装进线程池中，避免阻塞主事件循环。这种模式适合“先召回一批岗位，再逐个分析”的处理方式；代码中还保留了受控并发版本的设计，可通过 `Semaphore` 限制并发数，以适应模型限流与服务稳定性要求。

## 3.2 核心输入输出

| 项目       | 内容                                                         |
| ---------- | ------------------------------------------------------------ |
| 输入一     | `StudentProfile` 学生画像对象                                |
| 输入二     | `job_matching_profiles` 中的标准岗位画像向量索引             |
| 输入三     | 岗位知识库中的技能标准、能力标签、岗位要求等补充信息         |
| 中间结果一 | 经过知识库补全后的标准化学生画像                             |
| 中间结果二 | 经过硬条件过滤和多向量混合召回得到的候选岗位列表             |
| 中间结果三 | 经过 Adj-LightGBM 精排后的 Top-N 候选岗位                    |
| 输出一     | 推荐列表结果：`job_id + recall_score + rank_score + raw_data + deep_analysis + evidence` |
| 输出二     | 定向岗位诊断结果：针对单个岗位的结构化差距分析报告           |

当前推荐结果在结构上至少可包含岗位编号、召回分、精排分、岗位画像原始数据、深度分析结果，以及后续补充的知识库证据内容，便于前端直接渲染为推荐卡片、分析面板与差距解释视图。

## 3.3 流程 A：推荐式匹配（`POST /matching/jobs`）

该流程适用于“系统根据学生画像，一次性推荐 TopN 合适岗位”的场景，是第三步中最完整、最核心的主流程。

### 1. 接收学生画像并完成结构化校验

推荐式匹配的输入核心是 `StudentProfile`。学生的学历、专业背景、证书、实习时长、技能标签、工具标签、软素质评分、发展潜力评分、限制条件和实践经历都会被映射到固定字段中。这样可以保证补全层、召回层、排序层和分析层面对的是同一套结构稳定的画像输入，而不是前端任意拼接的自由文本。

### 2. 知识库检索与画像补全

在召回前，系统先利用岗位知识库中的标准岗位画像、技能标签体系和岗位要求词典，对学生画像进行补全与规范化处理。例如，将学生文本中的“会做前端页面”映射为更标准的技能表达，如 Vue、Element Plus、前端工程化等；将模糊的经历表达补全为项目经验、工具使用、场景能力等结构化特征。这样能够让后续匹配更稳定，也有利于提高召回命中率。

### 3. 硬条件预筛：先过滤明显不满足要求的岗位

系统将学生学历和实习月数映射为数值等级，再构造 Milvus 过滤表达式 `degree_level <= user_degree_level && exp_level <= user_exp_level`。因此，应届学生不会先被召回到大量“5年以上经验”“硕士及以上”的岗位，再依赖后续模型去否决，而是在召回前就完成第一层约束过滤。这个设计能显著降低后续召回噪声。

### 4. 多向量混合召回：从岗位知识库中找出候选岗位

对学生画像的四个维度，系统分别生成对应向量：

- `basic_info` → `vec_basic`
- `skills` → `vec_skills`
- `literacy` → `vec_literacy`
- `potential` → `vec_potential`

随后分别构建四个 `AnnSearchRequest`，使用相同过滤表达式在 Milvus 中并行召回候选岗位，再通过 `WeightedRanker` 做加权融合。当前默认权重可设置为 `[0.2, 0.5, 0.2, 0.1]`，即技能维度权重最高，基础要求和职业素养次之，发展潜力占比较小。该设置符合招聘场景下“优先看硬技能，再看基础条件与综合素质”的业务逻辑。

### 5. Adj-LightGBM 轻量精排：从“相关岗位”中筛出“更值得分析的岗位”

混合召回完成后，系统并不直接把全部候选岗位送入大模型，而是先使用 Adj-LightGBM 对候选集合做轻量精排。该阶段主要利用召回分、学历经验匹配度、技能覆盖度、项目经历相关性、岗位要求满足度等特征，对候选岗位进行快速排序。这样可以把真正更有潜力的岗位排到前面，减少低质量候选进入 LLM 分析阶段，从而在保证准确性的同时控制算力与调用成本。

### 6. LLM Top-N 深度重排：从“排序结果”升级为“职业判断结果”

经过 Adj-LightGBM 精排后，系统选取 Top-N 候选岗位交给 `CareerAnalystAgent.batch_analyze_async()`。该方法会先把学生画像序列化为 JSON 文本，然后为每个候选岗位创建异步分析任务，并通过 `asyncio.gather()` 并发执行。每个岗位内部调用 `_analyze_single_job_async()`，将“学生画像 + 岗位要求”拼接成 Prompt，请大模型输出结构化分析结果，重点包括：

- 是否可以投递 `can_apply`
- 综合匹配分 `score`
- 缺失关键技能 `missing_key_skills`
- 差距矩阵 `gap_matrix`
- 行动建议 `actionable_advice`
- 总体分析 `all_analysis`

这一阶段的本质，是把机器排序结果进一步提升为可解释的职业判断结果。

### 7. 知识库证据检索：为解释结论补充依据

当 LLM 完成深度分析后，系统会围绕该岗位进一步检索岗位知识库中的对应证据，例如岗位要求条目、核心技能说明、行业常用工具、能力标准或发展路径节点。通过把这些结构化证据与 LLM 的分析结论关联，系统能够输出“结论 + 依据”的双层解释结果。这样不仅有利于前端展示，也有利于用户理解“为什么不匹配”“差距具体体现在哪里”。

### 8. 结构化差距解释：输出最终推荐岗位

最后，系统将召回分、精排分、LLM 分析结果和知识库证据综合起来生成最终结果。结果不仅包含岗位推荐本身，还会补充以下结构化内容：

- 是否建议投递
- 当前匹配程度
- 关键差距项
- 差距所属维度
- 对应知识库依据
- 分阶段提升建议

这说明当前匹配流程采用的是典型的“**混合召回做粗选，学习排序做中排，大模型做精排，知识库做证据支撑**”的多阶段推荐架构。

------

## 3.4 流程 B：定向岗位匹配（`analyze_user_job_match`）

该流程适用于“用户点开一个具体岗位，查看自己与该岗位的详细匹配分析”的场景。

### 1. 读取双画像

`analyze_user_job_match()` 会先通过 `JobPortraitRepository.get_by_id(job_id)` 读取岗位画像，再通过 `StuProfileRepository.get_by_stu_id(user_id)` 读取学生画像。如果任一画像不存在，函数会直接抛出异常并返回明确错误，而不是生成模糊分析。

### 2. 序列化岗位画像与学生画像

系统使用 `_build_profile_json_text()` 将岗位画像和学生画像中关键字段序列化为 JSON 文本。这一步的目的，是减少模型对复杂嵌套结构的误读，确保 Prompt 输入格式稳定一致。

### 3. 构造 Prompt 与调用模型

该流程使用 `prompt_loader` 中定义的 `SYSTEM_PROMPT` 和 `USER_PROMPT`，再通过 `ChatPromptTemplate` 与 `ChatTongyi` 组成推理链。执行时优先走 `chain.ainvoke()`；若异步调用失败，再自动回退到 `asyncio.to_thread(chain.invoke, ...)`，保证单岗位分析链路在不同环境下都具备较好的可用性。

### 4. 知识库证据补充

在单岗位分析场景下，系统同样可以围绕该岗位对应的知识库内容进行检索，将岗位职责、技能要求、证书建议、典型能力路径等信息补充到分析结果中，使用户在查看单岗位诊断时，不仅能看到“差距结论”，还能看到“差距依据”。

### 5. JSON 提取与清洗

模型返回内容可能包含额外字符或 Markdown 包裹，因此系统会先调用 `_extract_json_text()` 尝试直接解析；若失败，再从返回内容中提取最外层 JSON 结构进行二次解析。如果仍然失败，则抛出“AI 返回内容无法解析为 JSON”的错误并记录日志。

### 6. Pydantic 校验与统一输出

解析成功后，系统会使用 `UserJobMatchResult.model_validate(parsed)` 强制校验结果格式，再返回统一的分析对象。最终输出中除了 `analysis` 本体外，还可回传 `job_profile_text`、`user_profile_text` 以及关联证据内容，便于调试、日志排查或前端辅助展示。

## 3.5 关键实现方法（建议评审重点关注）

### 1. 四维岗位画像匹配，而不是单向量相似度匹配

系统将岗位和学生都拆分为“基础要求、职业技能、职业素养、发展潜力”四个维度分别向量化，再通过混合检索加权融合，而不是把整份画像压缩为单个 embedding。这样得到的结果更细粒度，也更符合职业规划与岗位推荐场景。

### 2. 知识补全、规则过滤、语义召回分层执行

系统不是直接把原始用户数据拿去做向量搜索，而是先通过知识库做画像补全，再通过学历和经验规则做预筛，最后再进入语义召回。这体现了“知识增强 + 规则筛选 + 语义检索”的分层架构。

### 3. Adj-LightGBM 作为中间排序层，平衡效率与效果

在多向量召回与大模型深度分析之间加入轻量学习排序层，有助于把 LLM 的使用范围控制在更高质量的候选集合中。相比对所有召回岗位直接做深度分析，这种方式更节省算力，也更容易在工程上扩展。

### 4. LLM 输出结构化约束与容错

`CareerAnalystAgent` 要求模型返回 JSON，并在返回内容存在 Markdown 包裹时先清洗，再执行解析；若解析失败或接口异常，则返回兜底结果，避免接口直接崩溃。这说明深度分析层已经具备基本的结构化约束与失败恢复能力。

### 5. 知识库证据增强解释可信度

系统不是只输出“模型说你不匹配”，而是尝试把岗位知识库中的能力要求、技能条目、工具标准等内容作为证据回填到解释结果中。这使匹配结果不仅可看，还更可信、更容易被用户接受。

### 6. 并发分析与可扩展限流设计

当前版本通过 `asyncio.gather()` 并发分析多个岗位，提高整体吞吐；同时保留了用 `Semaphore` 控制并发数的扩展设计，以适应真实模型服务中的 QPS 限制与稳定性要求。

### 7. 召回、中排、精排分离，便于后续优化

该流程中 `recall_top_k`、`rank_top_k`、`final_top_k` 三层职责清晰：召回层保证候选覆盖，学习排序层提升候选质量，LLM 层负责深度判断与解释。这种分层设计天然适合后续继续加入行业偏好、地区限制、职业目标、用户意愿等更多排序特征。

### 8. 结果天然适合前端直接消费

推荐列表结果可以自带岗位基础信息、召回分、精排分、LLM 分析结果和知识库证据；单岗位诊断结果则可以直接返回总体评价、优势、不足、建议、风险点及其对应依据。这使前端无需自行拼接复杂解释逻辑，就能直接渲染为推荐页、详情页和报告页。

------

## 3.6 本阶段的核心功能点

1. **知识库驱动的画像补全**
   在匹配前先补全学生能力标签与岗位标准字段，提高后续匹配稳定性。
2. **岗位硬门槛拦截**
   在学历和经验层面先做预筛，减少明显不合适岗位进入召回集。
3. **多维度混合召回**
   不是只比技能，而是把基础要求、技能、素养、潜力四维一起纳入召回。
4. **Adj-LightGBM 轻量精排**
   在召回与大模型之间增加学习排序层，提高候选质量并控制 LLM 成本。
5. **LLM Top-N 深度重排**
   对高质量候选岗位做更深入的语义分析和差距判断，输出可解释结论。
6. **知识库证据检索与支撑**
   为差距解释、能力缺口和行动建议提供岗位知识依据，增强可信度。
7. **结构化差距解释与行动建议生成**
   不只告诉用户“匹不匹配”，还要说明“差在哪里、为什么差、怎么补”。
8. **单岗位深度诊断扩展能力**
   支持围绕某个具体岗位输出完整的匹配报告，适合作为职业规划详情页或面试准备分析页的后端能力。

------

## 3.7 第三步产出结果

完成第三步后，系统会生成两类可直接被前端消费的数据。

### 1. 推荐列表结果

每个岗位至少可包含：

- `job_id`
- 向量召回分 `recall_score`
- 轻量精排分 `rank_score`
- 岗位画像原始数据 `raw_data`
- 深度分析结果 `deep_analysis`
- 知识库证据 `evidence`

适合直接渲染为推荐卡片、匹配列表、岗位对比视图和推荐说明面板。

### 2. 单岗位诊断报告

面向某个具体岗位输出结构化匹配结论、差距解释、知识依据与改进建议，适合渲染到报告页、面试准备页或职业规划详情页。

## 3.8 本阶段的工程价值

从系统价值上看，第三步真正把前两步沉淀下来的岗位知识转化成了用户可感知的智能服务能力。前两步解决的是岗位数据接入、标准化与知识沉淀问题，而第三步回答的是用户最关心的几个问题：

**我适合什么岗位？我为什么适合或不适合？我缺什么能力？我该如何补齐差距？我是否值得投递？**

因此，这一阶段使整个系统不再只是一个静态的岗位知识库，而是一个能够提供**推荐、排序、分析、证据解释和行动建议**的职业规划智能体。
