# 岗位处理与匹配核心模块分析

本文档围绕项目中的三大核心能力展开：**岗位插入与清洗、岗位合并与画像生成、岗位匹配与分析**。这三部分并不是彼此独立的功能点，而是构成了一条完整的岗位智能处理链路：系统先把多来源、异构格式的原始岗位数据清洗并入库，构建统一的原始岗位数据层；再基于语义向量与聚类算法，将碎片化岗位归并为标准岗位画像，并沉淀为可复用的岗位知识库；最后结合学生画像，通过多向量混合召回与大模型深度分析，输出可解释的人岗匹配结果与改进建议。  

因此，这份文档是说明项目如何通过**数据清洗、语义聚类、结构化建模、向量检索和大模型分析**等技术手段，把原始招聘信息逐步转化为能够支撑职业规划、岗位推荐和能力诊断的智能系统能力。对比赛答辩而言，这三步也恰好构成了一条清晰的技术主线：**先有数据底座，再有岗位知识，再有用户侧智能匹配**。

------

## ⭐️ 第一步：岗位插入与岗位清洗 (Job Insertion & Cleaning)

**本部分使用的方法（一句话）：** 采用“多格式文件发现 + 分渠道清洗标准化 + 结构化映射 + 异步批量入库”的分层流水线，将来源不一、字段不齐、格式混乱的原始岗位数据转化为统一的原始岗位库。

**核心职责：** 该阶段承担整个岗位智能分析链路的“数据底座建设”任务。系统先从本地目录批量发现 Excel / CSV 岗位文件，再按文件类型分流到对应清洗器，对岗位名称、岗位描述、薪资、地点、日期等关键字段做规范化处理，随后映射为 `JobInfo` 实体并异步写入 MySQL 的 `job_original` 表。这个阶段的目标不是直接生成岗位画像，而是先构建一个**字段统一、噪声受控、可继续聚类与向量化**的原始岗位数据层，为后续“岗位合并”“岗位画像生成”“人岗匹配”提供稳定输入。

### 1.1 所用工具与技术栈

1. **Pandas 数据清洗引擎**
2. **Path / pathlib 文件系统遍历机制**
   `job_insert.py` 中通过 `Path` 与 `rglob()` 实现对指定目录的递归扫描。
3. **asyncio 并发调度与限流控制**
6. **可选的 LLM 岗位预过滤能力**
   Excel 清洗脚本中还实现了 `filter_computer_jobs_excel()`，它会先提取岗位名称、去重、再调用 `ChatTongyi + PydanticOutputParser` 对岗位名做“是否属于计算机 / 软件相关岗位”的批量判断，并按结果反向过滤原表。

### 1.2 数据模型与输入输出约束

1. **输入侧：多源异构岗位文件**
   当前导入主流程直接支持三类后缀：`.xls`、`.xlsx`、`.csv`。
2. **中间层：统一映射为 `JobInfo` 实体**
3. **输出侧：写入 `job_original` 原始岗位表**

### 1.3 实现思路与处理流程

1. **目录扫描与任务装配：批量发现待处理文件**
2. **按后缀路由到不同清洗链**
3. **Excel 清洗链：面向标准列式招聘表的规范化处理**
4. **CSV 清洗链：面向渠道字段杂乱场景的结构重组**
5. **字段级标准化方法：将脏数据转为可计算文本**
6. **实体映射：由表格行转为 ORM 对象**
7. **异步批量入库：构建原始岗位数据库**
8. **并发调度与资源释放：保障批量导入稳定性**

### 1.4 本阶段的核心功能点

1. **多格式岗位文件统一接入**
2. **岗位基础字段标准化治理**
3. **渠道异构字段向统一领域模型映射**
4. **异步批量入库与导入容错**

### 1.5 本阶段的工程价值

从系统架构角度看，第一步的价值不只是“把文件导进去”，而是在项目最前端建立一个可持续演进的数据治理层。它把原本零散、噪声大、字段不统一的岗位源数据，转化成结构统一、文本可计算、语义可聚类的原始岗位库。没有这一步，后续 HDBSCAN 聚类会受到岗位名脏词和描述噪声影响，LLM 画像提炼也会因为字段缺失与格式混乱而输出不稳定。因此，这一阶段本质上是在为整个职业规划系统构建“原始岗位标准层”。

------

## ⭐️ 第二步：岗位合并与画像生成 (Job Merging & Profile Building)

**本部分使用的方法（一句话）：** 采用“原始岗位向量同步 + HDBSCAN 语义聚类 + LLM 结构化提炼 + 岗位画像落库 + 向量索引同步”的多阶段流水线，将大量碎片化原始 JD 合并为可复用、可检索、可匹配的标准岗位画像。

**核心职责：** 本阶段是系统从“原始岗位数据层”走向“标准岗位知识层”的关键中枢。其核心目标不是简单做岗位去重，而是先基于原始岗位标题与描述构建语义表示，对岗位进行无监督聚类，再针对每个语义簇调用大模型提炼共性能力要求、职业技能、发展潜力与岗位属性，最终生成统一的标准岗位画像并写入 `job_profile` 表，同时同步至 Milvus 向量库，作为后续人岗匹配模块的岗位知识基座。

### 2.1 所用工具与技术栈

1. **HDBSCAN 密度聚类算法**
2. **本地语义向量模型 + 原始岗位向量库**
3. **PCA 降维 + 向量预处理**
4. **LLM 结构化岗位画像提炼**
5. **Pydantic / 结构化模型约束**
6. **asyncio 并发调度与受控限流**
7. **Milvus 标准岗位画像向量库**

### 2.2 数据模型与核心输入输出

1. **输入：原始岗位语义集合**
2. **中间结果一：岗位语义簇**
3. **中间结果二：结构化画像字典**
4. **输出一：标准岗位画像表 `job_profile`**
5. **输出二：原始岗位到标准岗位的映射关系**
6. **输出三：标准岗位画像向量索引**

### 2.3 实现思路与处理流程

1. **原始岗位向量同步：构建聚类输入层**
2. **文本特征构建：标题主导、描述辅助**
   原始岗位向量并不是直接拿 `job_desc` 做 embedding，而是通过 `build_text_feature()` 构造聚类文本：以清洗后的岗位标题为主，标题重复一次作为轻量加权，再追加岗位描述，且显式剔除部分噪声词和薪资表达。这样做能让聚类更加关注“岗位职责与能力模型”，而不是被薪资、福利或营销词干扰。
3. **HDBSCAN 聚类：形成候选岗位簇**
   聚类后的结果会按簇编号组织为岗位列表，`-1` 被视为噪声岗位；
4. **簇级并发分析：把岗位簇交给大模型归并**
5. **LLM 结构化提炼：从多个 JD 到一个标准岗位对象**
6. **岗位画像落库：建立标准岗位知识层**
7. **关系回写：打通原始岗位与标准岗位画像**
8. **标准岗位向量同步：生成匹配侧可检索索引**

### 2.4 本阶段的核心功能点

1. **原始岗位的语义归并**
2. **岗位画像自动生成**
3. **标准岗位命名与组织化沉淀**
4. **原始岗位到标准岗位的多对一映射**
5. **匹配侧岗位知识库初始化**
6. **工程化容错与并发保护**

### 2.5 本阶段的工程价值

从工程角度看，第二步真正完成的是一次“岗位知识抽象”。第一步得到的仍然是渠道化、碎片化的岗位记录，而第二步把这些记录提升为统一的标准岗位画像，并进一步变成可被向量检索、可被学生画像匹配、可被前端展示的岗位知识资产。换句话说，它解决的不是单纯的数据存储问题，而是岗位语义标准化、知识沉淀和后续智能推荐可用性问题。没有这一步，后续匹配系统只能在原始 JD 上做粗粒度相似度计算，很难得到稳定、可解释、可复用的人岗匹配结果。

## ⭐️ 第三步：岗位匹配与分析 (Job Matching & Analysis)

**本部分使用的方法（一句话）：** 采用“硬条件过滤 + 多向量混合召回 + LLM 深度重排 + 结构化差距解释”的多阶段匹配方法，输出既可排序、又可解释的人岗匹配结果与改进建议。

岗位匹配模块的目标并不是只返回一个相似度分数，而是构建一个完整的“**先筛选、再召回、再分析、再解释**”闭环：系统先依据学历与经验等硬性约束过滤掉明显不适配的岗位，再基于多维岗位画像向量完成候选岗位召回，随后调用大模型对候选岗位逐条做深度 Gap 分析和重排序，最终输出既可用于推荐列表展示、又能支持用户理解差距和制定提升路径的结构化结果。当前实现的核心重心在于**推荐式匹配链路**，而定向岗位诊断链路在你现有文档中已有说明。

### 3.1 所用工具与技术栈

1. **Pydantic 学生画像建模**
2. **Milvus 多向量混合检索**
   岗位召回并不是基于单一向量完成，而是依赖 `job_matching_profiles` 集合中的四个向量字段：`vec_basic`、`vec_skills`、`vec_literacy`、`vec_potential`。系统分别为学生画像的四个对应维度生成向量，再通过 `hybrid_search()` 发起多路 ANN 检索，并使用 `WeightedRanker` 进行加权融合。这种设计使召回结果不再单纯依赖某一项技能相似度，而是综合了基础要求、职业技能、职业素养和发展潜力四个层面。
3. **硬条件过滤机制**
   在正式召回前，系统会先把学历和经验映射为数值等级，例如博士=4、本科=2、专科=1，经验则按“应届 / 1-3年 / 3-5年 / 5年以上”粗粒度映射为 0~3，并生成过滤表达式 `degree_level <= user_degree_level && exp_level <= user_exp_level`。这意味着匹配流程并不是一开始就把所有岗位送入向量召回，而是先排除掉在学历或经验层面明显不满足的岗位，减少无意义召回与后续 LLM 成本。
4. **阿里云 Embedding + 岗位画像向量化**
5. **LLM 深度分析 Agent**
6. **定向岗位匹配分析器（对固定岗位进行匹配）**
7. **异步并发与线程池降级**

### 3.2 核心输入输出

| 项目     | 内容                                                      |
| -------- | --------------------------------------------------------- |
| 输入一   | `StudentProfile` 学生画像对象                             |
| 输入二   | `job_matching_profiles` 中的标准岗位画像向量索引          |
| 中间结果 | 经过硬条件过滤和多向量混合召回得到的候选岗位列表          |
| 输出一   | 推荐列表结果：`job_id + score + raw_data + deep_analysis` |
| 输出二   | 定向岗位诊断结果：针对单个岗位的结构化差距分析报告        |

当前可直接从代码确认的推荐结果中，至少包含 `job_id`、向量召回分 `score`、岗位原始画像 `raw_data`，以及后续由 LLM 补充的 `deep_analysis`。

### 3.3 流程 A：推荐式匹配（`POST /matching/jobs`）

该流程适用于“系统根据学生画像，一次性推荐 TopN 合适岗位”的场景，是当前第三步中最完整、最明确落地的主流程。

1. **接收学生画像并完成结构化校验**
2. **硬条件预筛：先过滤明显不满足要求的岗位**
3. **多向量混合召回：从岗位知识库中找出候选岗位**
4. **解析召回结果：形成候选岗位列表**
5. **LLM 并发深度分析：从相似岗位到可解释岗位**
6. **重排与截断：输出最终推荐岗位**

### 3.4 流程 B：定向岗位匹配（`analyze_user_job_match`）

该流程适用于“用户点开一个具体岗位，查看自己与该岗位的详细匹配分析”的场景。

1. **读取双画像**
2. **序列化岗位画像与学生画像**
3. **构造 Prompt 与调用模型**
4. **JSON 提取与清洗**
5. **Pydantic 校验与统一输出**

### 3.5 关键实现方法（建议评审重点关注）

1. **四维岗位画像匹配，而不是单向量相似度匹配**
2. **硬条件过滤与语义召回分层执行**
3. **LLM 输出结构化约束与容错**
4. **并发分析与可扩展限流设计**
5. **召回与精排分离，便于后续优化**
6. **结果天然适合前端直接消费**

### 3.6 本阶段的核心功能点

1. **推荐式岗位 TopN 输出**
   
2. **岗位硬门槛拦截**
   
3. **多维度相似度匹配**
   不是只比技能，而是把基础要求、技能、素养、潜力四维一起纳入召回。

4. **LLM 差距解释与行动建议生成**
   
5. **单岗位深度诊断扩展能力**

### 3.7 第三步产出结果

完成第三步后，系统会生成两类可直接被前端消费的数据：

1. **推荐列表结果**
   每个岗位至少包含：
   - `job_id`
   - 向量召回分 `score`
   - 岗位画像原始数据 `raw_data`
   - 深度分析结果 `deep_analysis`
     适合直接渲染为推荐卡片、匹配列表或岗位对比视图。
2. **单岗位诊断报告**
   面向某个具体岗位输出结构化匹配结论、差距解释与改进建议，适合渲染到报告页、面试准备页或职业规划详情页。该能力当前在你的文档中已有清晰定义。

### 3.8 本阶段的工程价值

从系统价值上看，第三步真正把前两步沉淀下来的岗位知识转化成了用户可感知的智能服务能力。前两步解决的是岗位数据接入、标准化和知识沉淀问题，而第三步回答的是用户最关心的几个问题：**我适合什么岗位、我缺什么、我该怎么补、我是否值得投递**。这使整个系统不再只是一个岗位知识库，而是一个能够提供推荐、分析和决策支持的职业规划智能体。
